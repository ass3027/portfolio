4-1 클로즈드 소스 바이너리 프로토콜 역공학 구현 ▾
문제
RPCN 서버는 공식 SDK가 없다. RPCS3 C++ 소스코드에서 15-byte 바이너리 헤더 + Protobuf 페이로드 구조를 분석하고 TLS/TCP 클라이언트를 직접 구현해야 했다.
15-byte Header <BHIQ> ├─ u8 pkt_type (Request=0 / Reply=1 / Notif=2) ├─ u16 command ├─ u32 total_size └─ u64 packet_id Payload: Simple → raw struct.pack LE integers Complex → u32 LE length prefix + Protobuf bytes
src/rpcn_client/client.py
def connect(self) -> int:
    ctx = ssl.create_default_context()
    ctx.check_hostname = False
    ctx.verify_mode = ssl.CERT_NONE  # RPCN 자체 서명 인증서

    raw = socket.create_connection((self.host, self.port), timeout=30)
    self._sock = ctx.wrap_socket(raw, server_hostname=self.host)
    hdr = self._recv_exact(HEADER_SIZE)
    pkt_type, _cmd, pkt_size, _pkt_id = struct.unpack(_HDR_FMT, hdr)

def _recv_reply(self) -> tuple[int, bytes]:
    while True:
        hdr = self._recv_exact(HEADER_SIZE)
        pkt_type, cmd, pkt_size, _pkt_id = struct.unpack(_HDR_FMT, hdr)
        payload = self._recv_exact(pkt_size - HEADER_SIZE)
        if pkt_type == PKT_NOTIF:   # 서버 푸시 알림 스킵
            continue
        return payload[0], payload[1:]

def transactional(fn):
    @wraps(fn)
    async def wrapper(*args, **kwargs):
        async with async_session_factory() as session:
            async with session.begin():   # 성공 시 commit, 예외 시 rollback
                return await fn(*args, session=session, **kwargs)
    return wrapper

┌─────────────── 조회/표시 (운행 리스트·잔여석·매진) ───────────────┐ │ 결과적 일관성 (staleness ≤ 2s) · Redis 단기 캐시 (single-flight) │ └───────────────────────────────────────────────────────────────────┘ │ ┌─────────────── 예매 확정 (락 임계영역) ───────────────┐ │ 강한 일관성 · Redis 원자 선점(SREM/SPOP) → DB 상태 전이 │ └─────────────────────────────────────────────────────────┘ DB = SoT (진실의 원천) · Redis = 빠른 게이트 부하 후 ReconciliationService(60s)가 DB로 드리프트 수렴
"남은 좌석 수가 화면마다 달라 보이는" 불일치는 버그가 아니라 조회 속도를 위해 의도적으로 감수한 트레이드오프 — 정확성이 필요한 예매 확정만 강한 일관성으로 격리했다. 그 경계를 어림짐작이 아니라 SLO 수치(S2b staleness ≤ 2s · S4 oversell 0)로 못박았다.
Key Challenges

핵심 기술적 도전

1 단일 원자 선점 — 1,000명 같은 좌석 경쟁 → 초과 판매 0건 ▾
문제
N명이 같은 좌석을 동시에 예매 시도할 때, 단 1명만 성공하고 좌석 재고가 정확히 −1 차감되어야 한다. 락을 여러 군데 흩뿌리면 경합 지점이 늘어 정합성이 깨진다.
해결
선점을 단일 원자 지점 = Redis Set avail:{schedule_id}에 모았다. 직접 선택은 SREM(반환 1 = 승리), 자동 배정은 SPOP. 승자만 DB 상태 전이(AVAILABLE→HELD→SOLD)를 수행하고, 패배 요청은 DB에 닿기 전 즉시 반려된다. 최종 방어선으로 DB @Version 낙관 락 + uk_active_seat 유니크 제약을 둔 이중 방어 구조다.
SeatPreemptionService — 원자 선점
// ① Redis 원자 선점: 반환 1 = 승리, 0 = 패배(즉시 409, DB 미접촉)
Long won = redis.opsForSet().remove("avail:" + scheduleId, seatId);
if (won == 0) throw new SeatAlreadyTakenException();   // 패배 999건을 DB 앞단에서 흡수

// ② 승자만 DB 상태 전이 — @Version 낙관 락이 최종 정합성 방어
seat.hold(entryToken, Duration.ofMinutes(5));          // AVAILABLE → HELD (TTL 5분)
seatRepository.save(seat);                             // uk_active_seat 로 중복 원천 차단
성과 (L1 실측)
1,000석 → 1,000명 동시 경쟁 → 성공 정확히 1건 → 999석 잔존. 초과 판매 0건 · 중복 0건을 k6 요약과 DB·Redis audit이 교차 검증(3회 일관). 진짜 1,000 동시(http_reqs=2000), 5xx 누수 0%.
2 조회 캐시 스탬피드 방어 — single-flight (E3) ▾
문제
운행 리스트·잔여석 조회는 모든 사용자의 진입점이라 읽기가 압도적으로 많다. 매 요청이 DB에서 재고를 SCARD로 집계하면 혼합 부하에서 조회 p95가 8~9초로 무너진다(캐시 off). 핫키 TTL 만료 순간 요청이 일제히 DB로 쏟아지는 스탬피드도 위험하다.
해결
Redis 공유 단기 캐시(TTL 1s)로 서빙하고, TTL 만료 순간의 스탬피드는 single-flight(winner 1개만 재계산·나머지는 결과 공유)로 막았다. 능동 무효화 대신 staleness를 TTL(≤1s)로 제한하고, 낡은 표시는 예매 시도가 SREM=0으로 자가교정(2-tier 계약)한다.
성과
핫키 조회 p95 8~9s → 26ms, 처리량 ~850 TPS → 2,497 TPS, dropped 0 (L3 3회 일관). 다만 캐시는 히트율 의존 — 저히트율 다중 키에선 DB 풀이 병목으로 복귀함을 함께 규명(정직한 한계).
3 비가시 입장 제어 — 대기열 없이 피크 흡수 (E2) ▾
문제
실제 KTX 앱처럼 대기 순번을 노출하지 않으면서 피크 트래픽을 견뎌야 한다. 입장 제어가 없으면 5xx 폭증이 아니라 무경계 열화(응답 p95 22s·VU 2,000 팽창·38k drop)로 시스템이 조용히 붕괴한다.
해결
Redis 활성자 카운터로 동시 활성 세션을 상한 K로 제한하고, 초과분은 보이는 대기열 없이 429/503 + Retry-After로 흡수한다(큐 번호 엔티티가 의도적으로 없음). 임계점 실험(L5b)으로 안전 처리량과 천장을 측정해 K=100을 확정했다.
성과
제어 off p95 22s·dropped 38k → 제어 on(K=100) 초과분 85.4%를 429로 흡수·dropped 0·reserve p95 ~28ms로 뒷단을 정격 보호 (L4 open-loop 3회).
4 Redis↔DB 드리프트 수렴 — DB를 SoT로 (오버셀 원천 차단) ▾
문제
빠른 게이트(Redis)는 언제든 틀릴 수 있다. 부하 중 Redis avail Set과 DB 상태 사이에 드리프트가 생기면, 방치 시 오버셀·재고 깨짐으로 번진다.
해결
ReconciliationService가 주기적으로(60s) 드리프트를 DB(진실의 원천)로 수렴시킨다. 방향은 비대칭 — DB엔 없는데 Redis에만 남은 stale 좌석 제거는 상시 안전하고, 그 반대(missing) 복원은 선점~커밋 사이 in-flight 오탐을 피해 grace(5분) 경과분만 되돌린다.
성과
"Redis=빠른 게이트, DB=SoT" 계약을 실제로 닫아 오버셀을 원천 차단. 정합성 자동 게이트(/internal/consistency)는 L6에서 timezone skew라는 환경 의존 실결함까지 자동 검출했다.

핵심 비즈니스 로직을 Service 계층에서 Entity로 이동하고, 상태 값을 기준으로 분기 처리하는 도메인 중심 구조로 재설계했다.
성과
비즈니스 로직 관리 대상 클래스 21개 → 1개로 축소, 변경 영향 범위 대폭 감소
2 GitLab CI/CD 최적화 — 빌드 시간 18분 → 6분 단축 ▾
문제
GitLab CI/CD 기반 개발 서버 배포 과정에서 빌드·테스트에 15~18분이 소요됐고, Docker 이미지를 외부 Docker Hub로 관리함에 따라 배포 보안 측면의 리스크도 있었다.
해결
GitLab CI 캐시를 활용해 dependencies 재다운로드를 제거하고, DEV/PROD 환경별로 중복 생성되던 Docker 이미지 빌드 구조를 단순화했다. 외부 Docker Hub 대신 사내 GitLab Image Registry를 도입해 내부 네트워크 기반의 이미지 관리 체계를 구축했다.
성과
CI/CD 빌드·배포 시간 18분 → 13분 → 6분 단축, Docker 이미지 관리 외부 의존성 제거로 배포 보안 강화
