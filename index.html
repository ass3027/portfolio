🎮 Tag2Now — 철권 태그2 실시간 대시보드

🐙 GitHub 🌐 서비스
개요

RPCS3 에뮬레이터로 TTT2를 즐기는 커뮤니티는 활성 유저 수, 매칭 상대, 랭킹 정보를 한눈에 볼 도구가 없었다.

RPCN은 오픈소스지만 게임의 호출 방식이 문서화돼 있지 않고 공식 SDK도 없다. Rust로 된 RPCN 서버를 직접 실행·분석해 호출 프로토콜을 역공학하고, 단위·통합 테스트로 검증한 Python 클라이언트를 구현하여 실시간 대시보드를 구축했다.

구현에 그치지 않고 AWS에 직접 배포하여 실제 도메인(match.tag2now.click)으로 서비스를 운영하는 것이 이 프로젝트의 목표였다. keyless CI/CD로 배포를 자동화하고, 오픈 카카오톡방의 실사용자 피드백을 받아 오류를 수정·개선하며 서비스를 유지했다.

핵심 기술적 도전

1RPCN 호출 프로토콜 역공학 & 검증된 클라이언트
문제
RPCN·RPCS3는 오픈소스지만 게임의 호출 방식(요청 순서·15-byte 바이너리 헤더 + Protobuf 페이로드)은 문서화되어 있지 않고 공식 SDK도 없다. Rust 서버를 직접 실행·분석해 프로토콜을 역공학해야 했다.
성과
서버 목록·방 검색·리더보드·스코어 조회 등 7개 커맨드를 재현·검증. 저수준 TLS/TCP 통신은 AI 협업으로 구현하고 단위·통합 테스트로 동작을 실측 검증. 명세된 struct는 이해·재사용하고 신규 기능에 맞게 수정.
2스냅샷 Diff 기반 매칭 탐지 (Phantom Room)
문제
매칭을 위해 방을 검색하고 있는 플레이어는 목록에서 사라져, 사용자 입장에선 활성 플레이어가 없는 것처럼 보였다.
성과
사라진 RANK_MATCH 방을 diff로 탐지해 "매칭 탐색 중" 플레이어를 가상 방(Phantom Room)으로 실시간 시각화 → 커뮤니티 체감 활성도 향상.
3비동기 선언적 트랜잭션 데코레이터
문제
SQLAlchemy 2.0 async 세션 관리 코드가 서비스 레이어 전반에 반복되었다.
성과
Spring의 @Transactional 패턴을 Python async에 적용한 @transactional 데코레이터로 세션 관리 코드를 제거, 예외 시 자동 롤백 보장.
4다중 데이터베이스 전략 & 계층적 캐싱
문제
방 목록(10s)·리더보드(60s)·통계(90일)·게시판(유연한 스키마) — 데이터마다 접근 패턴이 극명히 달랐다.
성과
Redis(TTL 캐시)·PostgreSQL(시계열 집계)·DynamoDB(교체 가능 게시판)를 데이터 특성별로 선별 적용, Ports & Adapters로 저장소 교체 가능.

🚄 KTX 티켓팅 — 대규모 동시성 성능·정합성 엔지니어링

개요

명절·연휴에 수만 명이 동시에 같은 좌석을 예매하려 할 때 발생하는 초과 판매(oversell)·중복 예매·좌석 재고 깨짐을 막고, 그 정합성과 성능을 부하 테스트와 Before/After 수치로 증명하는 것을 목표로 한 프로젝트다.

CRUD 기능 수가 아니라 N명이 한 좌석을 경합할 때 정확히 1명만 성공시키되(정합성), 그 정확성을 지키면서도 높은 처리량·낮은 지연을 함께 보장하는(성능) 엔지니어링이 핵심이다. 안전과 속도는 상충하기 쉬운데, 이 트레이드오프를 실측으로 풀어냈다. 7주 단계별 계획(P0~P6)으로 설계·구현하고, 부하 시나리오 L1~L6(9종) + 필수 실험 E1~E3의 Before/After를 모두 완료했다.

Key Challenges

핵심 기술적 도전 — KTX

1단일 원자 선점 — 1,000명 같은 좌석 경쟁 → 초과 판매 0건
문제
N명이 같은 좌석을 동시 시도할 때 단 1명만 성공하고 재고가 정확히 −1 되어야 한다. 락을 흩뿌리면 경합 지점이 늘어 정합성이 깨진다.
성과 (L1 실측)
선점을 단일 원자 지점(Redis Set SREM/SPOP)에 모으고 DB @Version 낙관락으로 이중 방어. 1,000명 경쟁 → 성공 정확히 1건, 초과판매·중복 0(3회 일관).
2조회 캐시 스탬피드 방어 — single-flight (E3)
문제
조회는 모든 사용자의 진입점이라 읽기가 압도적이다. 매 요청이 DB에서 재고를 집계하면 조회 p95가 8~9초로 무너지고, TTL 만료 순간 스탬피드도 위험하다.
성과
Redis 공유 단기 캐시(TTL 1s) + single-flight(winner 1개만 재계산)로 방어. 핫키 조회 p95 8~9s → 26ms, 처리량 ~850 → 2,497 TPS(저히트율 한계도 함께 규명).
3비가시 입장 제어 — 대기열 없이 피크 흡수 (E2)
문제
대기 순번을 노출하지 않으면서 피크를 견뎌야 한다. 입장 제어가 없으면 무경계 열화(p95 22s·VU 2,000 팽창·38k drop)로 조용히 붕괴한다.
성과
Redis 활성자 카운터로 동시 세션을 상한 K=100으로 제한, 초과분은 429/503+Retry-After로 흡수. 초과분 85.4%를 429로 흡수·dropped 0.
4Redis↔DB 드리프트 수렴 — DB를 SoT로
문제
